Abstract

This specification details a layered web API for an asynchronous key value store, in the style of the original localStorage.

1. Introduction

This section is non-normative.

The localStorage API is widely used, and loved for its simplicity. However, its synchronous nature leads to terrible performance and cross-window synchronization issues.

This specification proposes a new API, called async local storage, which is intended to provide an analogously simple interface, while being asynchronous. Along the way, it embraces some additional goals:

  • Layer on top of Indexed Database. This both fulfills the mandate of layered web APIs to build on top of existing platform features, and allows an upgrade path to full IndexedDB usage if a web developer outgrows the async local storage interface. [INDEXEDDB] [LWAPI]

  • Modernize the API surface. Modern key/value stores in the platform, such as the Cache or Headers APIs, have aligned around the operation names given by JavaScriptâs Map. We follow their example. As a bonus, this allows us to avoid the legacy named properties feature that the Storage interface uses.

  • Support isolated storage areas. localStorage requires careful namespacing of keys to use robustly in a multi-actor environment. Popular libraries meant to replace it, like localForage, have included a way to create new storage areas beyond the default one.

3. The StorageArea class

Upon evaluating the async-local-storage layered API module, the StorageArea class must be created in the current Realm. The result must be equivalent to evaluating the following JavaScript code, with the exception that the constructor, method, and getter bodies must behave as specified below instead of being the no-ops shown in this code block:

class StorageArea {
  constructor(name) {}

  set(key, value) {}
  get(key) {}
  has(key) {}
  delete(key) {}
  clear() {}

  keys() {}
  values() {}
  entries() {}

  get backingStore() {}
}

Each StorageArea instance must also contain the following internal slots:

Internal Slot Initial Value Description (non-normative)
[[Connection]] null An IndexedDB connection or null
[[ConnectionException]] null An exception encountered during the connection process or null
[[StoreName]] null A string or null

3.1. constructor(name)

  1. Let storageArea be this StorageArea object.

  2. Set storageAreaâs internal slots to the initial values indicated in Â§3 The StorageArea class.

  3. Set storageArea.[[StoreName]] to name.

  4. Let origin be the current settings object's origin.

  5. If origin is an opaque origin, throw a "SecurityError" DOMException.

  6. Let request be a new open request.

  7. Add a simple event listener to request for "upgradeneeded" that performs the following steps:

    1. Assert: storageArea.[[Connection]] is not null.

    2. Let database be the database associated with storageArea.[[Connection]].

    3. Assert: databaseâs upgrade transaction is active.

    4. If an object store name name already exists in database, throw a "ConstraintError" DOMException.

    5. Let store be a new object store in database.

    6. Set storeâs name to name.

  8. Run these steps in parallel:

    1. Let result be the result of opening a database, given origin, "async-local-storage", 1, and request.

    2. Queue a task to run these steps:

      1. If result is an IndexedDB connection, set storageArea.[[Connection]] to result.

      2. Otherwise, set storageArea.[[ConnectionException]] to result.

Modulo issues around tampered-with globals and prototypes, the above is roughly equivalent to the following JavaScript code: 
class StorageArea {
  // ...

  constructor(name) {
    this.#storeName = name;

    try {
      const openRequest = indexedDB.open("async-local-storage", 1);

      openRequest.onerror = () => {
        this.#connectionException = openRequest.error;
      };
      openRequest.onupgradeneeded = () => {
        openRequest.result.createObjectStore(name);
      };
      openRequest.onsuccess = () => {
        this.#connection = openRequest.result;
      };
    } catch (e) {
      this.#connectionException = e;
    }
  }

  // ...
}

3.3. get(key)

  1. Let storageArea be this.

  2. If storageArea does not have an [[StoreName]] internal slot, or storageAreaâs relevant realm is not the current realm, return a promise rejected with a TypeError exception.

  3. Let promise be a new promise.

  4. In parallel, wait for either storageArea.[[Connection]] or storageArea.[[ConnectionException]] to become non-null. Then, queue a task to run the following steps:

    1. If storageArea.[[ConnectionException]] is non-null, reject promise with storageArea.[[ConnectionException]], and abort these steps.

    2. Let connection be storageArea.[[Connection]].

    3. Assert: no running upgrade transaction is associated with connection.

    4. Assert: connectionâs close pending flag is unset.

    5. If connectionâs database does not contain an object store named storageArea.[[StoreName]], reject promise with a "NotFoundError" DOMException and abort these steps.

    6. Let store be the single object store in connectionâs database named storageArea.[[StoreName]].

    7. Let transaction be a newly created transaction with connection, "readonly", and Â« store Â».

    8. Set transactionâs cleanup event loop to the current event loop.

    9. Let handle be a new object store handle whose object store is store and whose transaction is transaction.

    10. Let range be the result of converting a value to a key range with key and the null disallowed flag set. If this throws an exception, reject promise with that exception and abort these steps.

    11. Let request be the result of asynchronously executing a request given:

      • A source of handle

      • An operation consisting of running the steps to retrieve a value from an object store given the current Realm, store, and range.

    12. Add a simple event listener to request for "success" that resolves promise with requestâs result.

    13. Add a simple event listener to request for "error" that rejects promise with requestâs error.

  5. Return promise.

Modulo issues around tampered-with globals and prototypes, and using a not-shown-here utility waitForConnectionOrConnectionException(), the above is roughly equivalent to the following JavaScript code: 
class StorageArea {
  // ...

  get(key) {
    try {
      this.#storeName;
    } catch (e) {
      return Promise.reject(e);
    }

    return waitForConnectionOrConnectionException().then(() => {
      if (this.#connectionException) {
        throw this.#connectionException;
      }

      return new Promise((resolve, reject) => {
        const transaction = this.#connection.transaction(this.#storeName, "readonly");
        const store = transaction.objectStore(this.#storeName);
        const request = store.get(key);

        request.onsuccess = () => resolve(request.result);
        request.onerror = () => reject(request.reject);
      });
    });
  }

  // ...
}

3.11. Supporting operations

To add a simple event listener, given an EventTarget target, an event type string type, and a set of steps steps:

  1. Let callback be a new callback function whose callback context is the current settings object and whose object reference is a new JavaScript function that performs the steps given by steps. Other properties of the function (such as its name and length properties, or [[Prototype]]) are unobservable, and can be chosen arbitrarily.

  2. Let listener be a new event listener whose type is type, callback is callback, capture is false, passive is false, and once is false.

  3. Append listener to targetâs event listeners.
